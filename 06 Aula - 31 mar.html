O que é Normalização?

📐

Definição

Processo de organizar atributos e relações de um banco de dados para reduzir redundâncias e melhorar a integridade dos dados.

🎯

Objetivos

  • Eliminar dados duplicados
  • Garantir dependências lógicas
  • Facilitar manutenção
  • Prevenir anomalias
⚠️

Anomalias evitadas

  • Inserção: não conseguir inserir sem outro dado
  • Exclusão: apagar e perder outra informação
  • Atualização: ter de corrigir em N lugares
🏛️

Formas Normais

  • 1FN — Atomicidade
  • 2FN — Dep. total na PK
  • 3FN — Sem dep. transitiva
  • FNBC, 4FN, 5FN…

1ª Forma Normal
04 / 08

1FN — Primeira Forma Normal

REGRA

Todos os atributos devem ser atômicos (indivisíveis). Sem múltiplos valores em uma célula e sem grupos repetidos.

⚠️ CLIENTE — telefone multivalorado

Um cliente pode ter vários telefones. Armazenar em um campo viola a atomicidade: "(11)9999-0000, (11)8888-1111"

⚠️ LIVRO — status sem domínio controlado

Status como string livre ("disponivel", "Disponível") gera inconsistências e erros de digitação.

✅ Solução: tabelas separadas TELEFONE e STATUS referenciadas por FK.
TABELAS GERADAS NA 1FN
TELEFONE (nova)
  • PKFK id_cliente
  • PK seq_telefone
  • telefone
STATUS (nova)
  • PK id
  • nome
TELEFONE usa chave composta (id_cliente + seq_telefone) — permite N telefones por cliente sem precisar de um ID surrogate adicional.
2ª Forma Normal
05 / 08

2FN — Segunda Forma Normal

REGRA

Deve estar na 1FN e todo atributo não-chave deve depender da chave primária completa. Proibida dependência parcial.

ANÁLISE — CLIENTE (PK simples: cpf)
cpf
→
nome
cpf
→
rua, bairro, cep, cidade, estado, pais
⚠️ Endereço não depende diretamente do CPF

Rua, bairro, cidade etc. pertencem a um endereço, que por sua vez depende do CEP. Se dois clientes moram no mesmo CEP, os dados se repetem.

✅ Solução: tabela ENDERECO com chave composta (cpf_cliente, seq_endereco) — permite múltiplos endereços por cliente sem redundância.
TABELAS APÓS 2FN
CLIENTE
  • PK cpf
  • nome
ENDERECO (nova)
  • PKFK cpf_cliente
  • PK seq_endereco
  • cep
  • logradouro
  • numero
  • complemento
  • bairro
  • FK id_cidade
id_cidade referencia a tabela CIDADE — que será criada na 3FN para eliminar a dependência transitiva cidade → estado → pais.
3ª Forma Normal
06 / 08

3FN — Terceira Forma Normal

REGRA

Deve estar na 2FN e nenhum atributo não-chave deve depender de outro atributo não-chave (sem dependência transitiva).

DEP. TRANSITIVA — ENDEREÇO
cpf_cliente
→
id_cidade
→
id_estado → id_pais
⚠️ Hierarquia geográfica encadeada

cidade determina estado, e estado determina pais. Guardar tudo em ENDERECO repete o nome da cidade para cada CEP daquele município.

DEP. TRANSITIVA — LIVRO
cod_barras
→
nome_autor, nome_editora
⚠️ Autor e editora como texto livre

Guardar "J.K. Rowling" em cada livro dela repete a string. Atualizar o nome exigiria alterar N registros.

TABELAS CRIADAS NA 3FN
PAIS (nova)
  • PK id
  • nome
  • cod_pais
ESTADO (nova)
  • PK id
  • nome
  • uf
  • FK id_pais
CIDADE (nova)
  • PK id
  • nome
  • FK id_estado
EDITORA (nova)
  • PK id
  • nome
AUTOR
  • PK id
  • nome
STATUS
  • PK id
  • nome
Hierarquia CIDADE → ESTADO → PAIS em 3 tabelas independentes. Cada uma pode ser consultada, atualizada e expandida sem afetar as demais.

Comparativo das Formas Normais

2FN 1FN + dependência total na PK Dependência parcial de atributos não-chave ENDERECO separado com PK composta (cpf_cliente, seq_endereco) — suporta múltiplos endereços por cliente
